# 2주차 과제 회고

## 2주차 과제 설명

2주차는 1주차 과제 코멘트를 바탕으로 리팩토링을 진행했고, 1주차 과제에서 미흡했던 기능들을 추가하면서 멀티 스레드를 구현했다.

---

### Book 클래스

우선, `Book` 클래스의 상속 구조를 바꾸었다. 기존에 책의 카테고리와 책의 형태를 2차 상속으로 구현했었지만, 책의 형태만 자식 클래스로 남겨두고, 책의 카테소리는 `Book` 클래스의 `eunm` 타입으로 구성하였다.  하지만 전체적으로 출력하는 메소드가 너무 많고, 다른 부분에 집중 하다 보니 `EBook`클래스 기능 구현을 마치지 못한것이 아쉽게 느껴진다.

- `BookName`
    - 책 이름을 저장하는 변수
    - 자식 클래스가 사용할 수 있도록 `protected` 로 선언
- `BookForm`
    - 책의 형태를 저장하는 변수
    - 실제 값은 자식 클래스에서 설정
- `public enum BookCategory`
    - 책 카테고리를 관리하는 enum 타입
    - 문학, 경제, 과학 같은 분류를 상수 형태로 저장
    - 한국어로 반환할 수 있도록  `getKorean()` 구현
- `public String BookGetName()`
    - 현재 책 이름을 반환하는 메서드
    - 외부에서 BookName에 접근할 필요가 없도록
- `public Book(String BookName, String bookform)`
    - 책 이름과 형태를 초기화하는 생성자
    - 자식 클래스 객체가 생성될 떄, 값을 받아옴.
- `public void BookNamePrint()`
    - 책 이름을 출력하는 메서드
- `public void BookStatusPrint()`
    - 책 형태 정보를 출력하는 메서드
    - 이때 괄호를 닫지 않은 이유는 종이책에서 잔여책을 출력하기 위함
- `public boolean isLoanAvailable()`
    - 책 대출 가능 여부를 반환하는 메서드
    - 현재는 EBook 클래스에서는 필요가 없지만, 이후에 EBook 의 기능이 확대( 다운로드 링크를 생성하고 일정 기한이 지나면 접근 불가능 하도록.. ) 된다면 필요한 기능이기 떄문에 Book 클래스에 구현해두었다 ( 오버라이딩 해서 쓰기 → 종이책은 이미 쓰고 있음 )
- `public void BookLoan()`
    - 책 대출 상태를 출력하는 메서드
- `public void BookReturn()`
    - 책 반납 상태를 출력하는 메서드

### EBook 클래스

`EBook`은 전자책이라는 특성을 살려, 책을 다운로드 하는 방식으로 생각해보았다.

- `static final String BookForm = "전자"`
    - 전자책 형태를 고정값으로 저장하는 상수
- `String BookDownload = "책 다운링크"`
    - 전자책 다운로드 링크를 저장하는 변수
- `boolean isDownload = false`
    - 다운로드 가능 여부를 저장하는 변수
- `public EBook(String BookName)`
    - 전자책 이름을 초기화하는 생성자
    - 부모 클래스(`Book`) 생성자를 호출해 책 이름과 형태를 전달
- `@Override public void BookLoan()`
    - 전자책 대출 기능을 재정의한 메서드
    - 현재 실행 중인 스레드 이름을 출력하고,
    - 다운로드 권한을 부여한 뒤 부모 클래스의 대출 메서드를 호출
- `@Override public void BookStatusPrint()`
    - 책 상태 출력 기능을 재정의한 메서드
    - 부모 클래스의 상태 출력 뒤 괄호로 닫음.
- `void BookAuthority()`
    - 전자책 다운로드 권한을 부여하는 메서드
    - 다운로드 가능 상태를 `true`로 변경한다.
    - 이후 일정 시간이 지나면 권한을 제거하는 기능을 추가할 수 있다.
    - 권한을 제거하는 기능은 생성한 다운로드 링크 객체를 버리는 방식으로 진행하면 될 것 같다.

### PaperBook

`PaperBook` 클래스는 종이책이라는 특성을 살려서 책 위치와 책 개수를 필드로 두었다. 

- `static final String BookForm = "종이"`
    - 종이책 형태를 저장하는 상수
- `String BookLocation`
    - 도서관 내 책 위치를 저장하는 변수
- **`AtomicInteger TotalBooks` (동기화)**
    - 현재 남아있는 책 권수를 저장하는 변수
- `public PaperBook(String BookName, int totalBooks, String bookLocation)`
    - 종이책 이름, 책 개수, 위치를 초기화하는 생성자
- `void BookDecrease()`
    - 책 권수를 1 감소시키는 메서드
- `void BookIncrease()`
    - 책 권수를 1 증가시키는 메서드
- `void BookCoutPrint()`
    - 현재 남은 책 권수를 출력하는 메서드
- `@Override public void BookStatusPrint()`
    - 책 형태와 남은 권수를 함께 출력하는 메서드
- `void BookLocationPrint()`
    - 책 위치를 출력하는 메서드
- `@Override public boolean isLoanAvailable()`
    - 대출 가능한 책이 남아있는지 확인하는 메서드
    - 책 권수가 1권 이상이면 `true`를 반환
    - 아니라면 `false` 반환
- **`@Override synchronized public void BookLoan()` (동기화)**
    - 종이책 대출 기능을 재정의한 메서드
    - 대출 가능 여부 확인 후 책 권수를 감소
    - 책 위치 출력, 잔여책 권수 출력
- **`@Override synchronized public void BookReturn()`** **(동기화)**
    - 종이책 반납 기능을 재정의한 메서드
    - 책 권수를 증가
    - 잔여책 권수 출력

#### 동기화

책을 대출하거나 반납하는 상황에서 **`TotalBooks`가 증가 하거나 감소하게 된다. 그래서 원자성을 보장하기 위해 `AtomicInteger` 로 선언하였다.**

그리고 대출을 하는 과정에서 `BookLoan()` 함수에 원래는 대출을 실제로 수행하는 기능만 넣어두었다 ( 책 권수를 감소 하기만 했음 ) 그런데 대출하는 스레드를 여러개 둬서 실행하게 되면 검증을 했을 때는 대출이 가능했지만 실제 대출이 되는 상황에서는 책 권수가 0이 된 상태에서 책 권수를 감소해서 -1이 되는 상황을 발견했다.

또한, 사용자들이 동시에 모두 `BookLoan()` 함수에 들어가게 되면 대출하는 기능이 꼬이게 되는 상황이 발생한다. 아마 반납도 마찬가지 알것이다.

### BookList

`BookList` 클래스에서는 책들의 객체 생성 기능 구현을 하였다. 우선, 책들이 카테고리로 나뉘어져 있다보니 `Map` 자료구조를 활용해서 `Map<Book.BookCategory, List<Book>>` 타입으로 선언 하였다.  `Key` 값은 `Book` 클래스의 `BookCategory` 로 두고, `value` 값을 `Book` 객체를 리스트 형식으로 넣어두었다. List 타입을 사용한 이유는 책의 위치를 001,002 와 같은 방식으로 표현하였기 때문에 순서가 보장되면 더 좋을거 같다고 생각했다. 

이렇게 구성을 하다보니 카테고리 별로 책을 나눌 수 있게 되었고, `List<Book>` 타입이기 떄문에 `PaperBook,EBook` 에 상관하지 않고 객체를 생성할 수 있었다. 

`BookList`를 구현하면서 `List` 와 `Map`에 대해 더 잘 이해할 수 있었다.

#### 아쉬운 점

객체를 `Book` 타입으로 관리하면 `PaperBook`, `EBook`을 하나의 리스트에서 다룰 수 있어서 편하지만, 특정 기능이 한 자식 클래스에만 필요한 경우에는 문제가 생긴다.

예를 들어 반납 기능은 종이책에는 필요하지만, 전자책에는 필요하지 않다. 그런데 `Book` 부모 클래스에 `BookReturn()` 메서드를 만들어두면 전자책도 그 메서드를 가지게 된다. 이러면 전자책 입장에서는 필요 없는 기능을 상속받는 구조가 된다.

반대로 `BookReturn()`을 `PaperBook`에만 두면, 리스트에서는 `Book` 타입으로 꺼내기 때문에 바로 호출할 수 없고 `instanceof`로 종이책인지 확인한 뒤 형변환을 해야하는 문제가 생긴다.

### LibraryReturn 클래스

`LibraryReturn` 클래스에서는 도서관 시스템 업무 중 반납 기능을 수행하게 한다. 

하지만 사실 반납 기능은 종이책에만 필요한 기능이다. ( 전자책은 사실상 기한이 정해져 있고, 그 안에 보면 되는것이라서… ) 근데 `List<Book>` 타입이라, 종이책과 전자책 상관없이 리스트에 담겨 있어 종이책인지 아닌지 확인하는 로직을 넣었다. 

#### 예외 처리

책 이름이 리스트에 있는 값에 들어있지 않다면 반납에 실패하는 로직을 구성했는데,처음에는 이것을 if가 아닌 `try-catch`를 사용해서 구현해보려고 했지만, 책 이름을 찾지 못하는 상황을 예외로 만들어서 발생 시키는 것 자체가 억지로 예외를 만든 느낌이 들어서, 그냥 if를 사용하여 조건 분기를 만드는게 더 나을 것 같다는 판단을 했다.

### LibraryLoan 클래스

`LibraryLoan` 클래스에서는 도서관 시스템 업무 중 대출 기능을 수행하게 한다. 

대출할 책 개수를 입력 받고, 대출할 책 이름을 받아 현재 책이 존재하는지 비교하고 존재한다면 대출이 가능하게 동작한다. 대출은 총 3권 까지만 가능하고, 대출은 전자책,종이책 모두 가능하다.

대출이 가능한지 알아보기 위해 Book 클래스의 `isLoanAvailable()` 를 호출 하였다. Book 타입이긴 하지만, 현재 객체는 `PaperBook`과 `EBook` 클래스 타입으로 되어 있기 때문에 각 `PaperBook`과 `EBook` 클래스 타입에 정의 되어 있는 `isLoanAvailable()` 호출 하게 된다. → 이후 나올 `BookLoan()`도 동일한 방식으로 동작한다.

대출할 책이 모두 결정 되면 대출이 시작된다. `BookLoan()` 메서드로 대출 로직이 실행된다.

이후 대출이 모두 완료되면 반납 안내에 대한 내용이 출력된다.

### Verify 클래스

- `static final String LibraryCode = "1234"`
    - 도서관의 인증코드를 저장하는 LibraryCode 상수
- `public void VerifyCode()`
    - 도서관 인증 코드 검증 기능 수행

#### 예외 처리

try-catch 로 예외처리를 진행하였다. code를 숫자 4자리 이지만 문자열 형태로 입력 받고 있어서 단순히, `InputMismatchException` 로만 한다고 하면 인증 코드의 형태를 다르게 입력하던, 형태는 맞지만 코드와 다를 경우 모두 똑같은 예외로 빠지게 되는 상황이 발생했다. 그래서 4자리 숫자인지 형태를 먼저 확인하고 그 형태가 아닐 경우 throw 를 사용하여 `InputMismatchException`를 발생 시키고, “도서관 코드는 4자리 숫자” 라는 문구를 출력한다. 4자리 숫자는 맞지만 코드와 다를 경우 throw를 사용하여 `IllegalArgumentException` 를 발생시켜 “도서관 코드가 아닙니다” 라는 문구를 출력한다.

### Main 클래스

#### 멀티 Thread 생성

시간을 정해두고, 랜덤으로 책 대출이 실행되고, 책 반납이 실행되는 스레드를 구현했다. `loanthread` 와 `returnthread` 모두 runnable 로 실행 동작을 따로 구현하고, thread 는 실제 스레드 생성만 하도록 하였다. 

`BookLoanTask` 는 생성된 객체의 책을 모두 리스트에 담고 그 리스트 중에서 랜덤으로 대출이 진행되도록 하였다. 입력을 하기에는 무리 인거 같아서, 대출 기능만 수행하도록 `BookLoan()` 메소드를 실행하였다.

`BookReturnTask` 는 생성된 객체의 책을 모두 리스트에 담고, 그 리스트 중에서 종이책 객체 만 랜덤으로 반납이 진행되도록 하였다. 대출과 마찬가지로 입력을 하기에는 무리 인것 같아서, 반납 기능만 수행하게씀 `BookReturn()` 메서드를 실행하였다.

또한, 이 프로그램이 실행되면서 모든 메서드가 같은 객체를 공유해야하기 때문에, `BookList`를 이용한 책 객체 생성은 `Main` 클래스에서 한번만 실행했다.

## 아쉬운 점

2주차에 수업에서 많은 것을 배웠지만 적용하지 못한것이 아쉬웠다. 과제를 하면서 배운 것을 적용해보고 싶은데, 정확히 어디가 문제인지 제대로 파악을 못하는것 같아서 그냥 개념을 하나씩 다시 보면서 이게 코드에 적용이 될 수 있는지 파악을 하면서 해야할지 고민이 되었다.

그리고 클래스 다이어 그램이나 글로 적으면서 코드의 구조를 고민하고 코드를 작성해도, 작성하고 테스트를 하는 과정에서 코드가 자꾸 덧붙여 지면서 원래 생각했던 구조가 꺠지기도 하고, 코드 자체가 너무 지저분 해지는 느낌을 받았다..

또한 동기화를 해결하긴 했지만, 대출이 되거나 반납이 되는 기능 말고도 더 다른 흐름이 있을 수도 있을거 같은데, 딱히 상호작용할만한 기능이 생각이 안나 이렇게 구현하게 되었다. ( 수업시간에 배운 동기화 개념을 좀 더 사용하고 싶었지만, 딱히 공유 자원이라고 하는것이 `TotalBooks` 밖에 떠오르지 않았다… )
